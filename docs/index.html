Specification language · JVM-first

You write the contract. The compiler writes the code.

Declare the shape of a program — its types, signatures, and guarantees — and specify each behaviour as intent, as tests, or both. A language model synthesises the implementation; the compiler proves it against everything you wrote and emits ordinary, verified code for your target platform.

The model proposes. The compiler disposes. Code that cannot be proven against its specification never ships.

public User register(Email email, Password pw) {
  if (users.existsByEmail(email))
    throw new DuplicateEmail(email);
  var hash = BCrypt.hashpw(
      pw.value(), BCrypt.gensalt());
  return users.save(

}

async register(
    email: Email, pw: Password): Promise<User> {
  if (await this.users.existsByEmail(email))
    throw new DuplicateEmail(email);
  const hash = await bcrypt.hash(pw.value, 12);

}
# 4 contracts ✓   1 example ✓
def register(
    self, email: Email, pw: Password
) -> User:
    if self.users.exists_by_email(email):
        raise DuplicateEmail(email)
    hashed = bcrypt.hashpw(
        pw.value, bcrypt.gensalt())
    return self.users.save(
        User(email, Secret.of(hashed)))

Two layers, side by side

One layer you own. One the model realises.

Every declaration is made of two layers. The hard layer is deterministic and reviewable like any source code. The soft layer is where prose and tests buy you brevity — but it is never trusted blindly.

Hard

The shape and the guarantees

Types, fields, signatures, contracts, policies. Diffable, reviewable, and enforced by the compiler — not by convention.

Written by you · enforced by the compiler

Soft

What a body should do

intent prose and example / spec tests that describe behaviour. Verified against the hard layer before it becomes your program.

Specified by you · realised by the model

Verification, not vibes

Nothing reaches the artifact without passing its contracts.

When SkyLang synthesises a body it runs every example and checks every ensures and raises. A candidate that violates the specification is not quietly accepted — it is thrown out and regenerated, and if it still cannot be proven, the build stops and shows you the clause that failed. The same check applies to hand-written code.

candidate ensures result.stock ≥ 0  ✗
verdict rejected · regenerate
candidate all contracts & examples  ✓
verdict frozen · shipped

The tests write themselves

Every example and contract becomes a real unit test.

You never write test scaffolding. Each example compiles to a unit test in your target's framework that builds the arguments, calls the method, and asserts the expected result — and every ensures clause rides along as an assertion inside it. This generated suite is exactly what the compiler runs to accept or reject a synthesized body, and it ships in the emitted project for you to keep and re-run.

shop.sky · one contract, one example
service Catalog {

  restock(p Product, units Int) -> Product
    intent   "Increase the product's stock by units."
    requires units > 0
    ensures  result.stock == p.stock + units
    example  restock(Product(1, "Notebook", 5), 3)
             -> a Product with stock 8
}

The interface layer

Pages, flows, and components are contracts too.

The interface is declared, not assembled: what a page shows, which actions it offers, how it appears, and what you expect of it. The compiler synthesises the page, deploys the application into an embedded container, and proves every expectation against the rendered page — real markup, real navigation, offline. Flows bind multi-page journeys to typed steps, components are reusable widgets with their own expectations, and sign in is a declaration that compiles onto OpenID Connect.

shop.sky · the page is the spec
page ProductList at "/products" {
  shows   Catalog.all() as a sortable table of (name, stock)
  action  "Restock" on row -> Catalog.restock(row, ask Int)
  action  "Sign in" -> sign in then page Dashboard
  expect  table has columns (name, stock)
  appears the sign-out control when signed in
}

flow Checkout {
  step Cart -> page CartPage
  step Pay  -> CheckoutService.place(the cart)
  on PaymentFailed -> back to step Pay
}

Your own vocabulary

Declare an annotation once. It drives behaviour wherever it lands.

An annotation is a named bundle of intent and expectations — optionally parameterised — that attaches to an entity, a service, a method, a page, or a component. Its substituted prose steers every implementation realised under the target and joins its freeze hash: annotate a service and every body it owns is regenerated and proven against its contracts again; sharpen the annotation itself and everything it touches follows. Built-ins ride the same surface — @scope(session) gives a service one bean instance per visitor.

shop.sky · your vocabulary, checked like code
annotation memory_optimized {
  intent "Prefer streaming, constant-memory
          implementations; never materialize
          whole collections."
}

annotation databasebacked(store Text) {
  intent "Persist through the {store} store."
  expect every read and write goes through {store}
}

@memory_optimized
service Catalog {
  all() -> [Product]  intent "Every product."
  restock(id Int, units Int) -> Product
    intent "Increase stock."
}
attach→re-prove

The compilation pipeline

From spec to artifact — the model lives only here, at build time.

Production is an ordinary artifact of the target platform. No model on the runtime path, ever.

01

Parse & type-check

The hard layer only — refined types, effect sets, contract well-formedness. No model involved. This is what sky check runs.

02

Synthesise & verify

For each body: reuse a frozen version, or call the model and verify the candidate against your contracts and every active policy.

03

Freeze

Write the accepted source plus a content hash into sky.lock. Later builds read the frozen body — no model, no network.

04

Emit

Materialise a conventional project for your profile and hand it to the standard toolchain. Real code you get to keep.

Reproducible builds, with a model in the loop

A green build today is byte-identical tomorrow.

The obvious objection: if a model writes the code, how is my build reproducible? SkyLang answers with freezing. The first build verifies each body and records it in sky.lock, committed to version control. Every later build reads the frozen, already-proven source — offline and identical for everyone. Change one intent or contract, and only that method regenerates.

The toolchain

A small set of verbs for the red → generate → green loop.

sky onboard

Point the compiler at OpenAI or Anthropic — pick a model and reasoning effort, once.

sky check

Type & contract check only. Fast, offline, no model.

sky build

Synthesise and verify unfrozen bodies, then emit the artifact.

sky run

Build, then bring the application up in a browser — the artifact itself, running.

sky preview

Serve your views live in a browser studio, re-rendering as you edit.

sky tdd

Watch mode: on a failing test, regenerate just that method until it's green.

sky why <m>

Show a method's intent, contracts, frozen body, and verification report.

sky freeze

Force-regenerate and re-verify every body, rewriting sky.lock.

sky test

Run every example, spec, and ensures clause as a suite.

sky clean

Delete the build directory; sky.lock re-materialises it next build.
